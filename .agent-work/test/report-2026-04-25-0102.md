### 🔍 P2-NEW-002 — 双重余额恢复（重新审查）

**文件**: `table-engine/table-manager.service.ts` lines 45-99, `wallet/wallet.service.ts` lines 159-198

**问题描述**：崩溃重启后，`onModuleInit` 清理 WAITING 桌的残留座位并调用 `resetBalanceAndUnfreeze`。但如果玩家在 **DEALING/PREFLOP/FLOP/TURN/RIVER/SHOWDOWN** 状态的牌桌中崩溃（非 WAITING），则不会触发此清理逻辑。`getTable` 会从 Redis/SQLite 恢复牌桌状态，但玩家余额可能处于不一致状态（部分筹码在牌桌 stack 中，部分在 frozenChips 中）。

**现有防护分析**：

- `resetBalanceAndUnfreeze` 使用 `$transaction` 原子操作，防止了 balance 恢复但 frozenChips 未清除的 race condition
- `cleanupOfflineResidueOnStartup` 只清理 **WAITING** 状态牌桌的残留座位（line 49-52: `state: GameStage.WAITING`）
- DEALING+ 状态牌桌的玩家不在清理范围内

**关键代码**（table-manager.service.ts lines 49-52）:

```typescript
const staleWaitingTables = await this.prisma.table.findMany({
  where: { state: GameStage.WAITING },
  include: { players: true },
  ...
});
```

**风险评估**：

- 如果服务器在牌局进行中崩溃（非 WAITING 桌），重启后：
  1. 玩家余额 frozenChips 未被恢复
  2. `unfreezeBalance` 不会被调用（因为玩家仍在活跃牌桌中）
  3. 玩家看起来"在线"但无法操作
- 但玩家重新 join 时，`processAction` 会检查 `activePlayer.stack` 与 `getUserAvailableBalance` 的差异

**结论**: 🟡 P2-NEW-002 风险存在但非严重，因为活跃牌桌玩家不会被意外清理。真正风险是崩溃后玩家被锁定在牌桌上无人唤醒——这属于运维/高可用范畴。

**建议**: 增加 `TableManagerService` 单元测试，覆盖服务器重启后牌桌恢复场景。

### 🔍 P2-NEW-001 二次审查 — raise 保护仍存在边界 case

**文件**: `table-engine/table-player-ops.ts` lines 316-322

**当前代码**:

```typescript
case 'raise': {
  if (
    this.table.calledAllIn !== null &&
    amount >= this.table.calledAllIn
  ) {
    return false;
  }
```

**漏洞场景**：

- Alice all-in 100，Bob call（calledAllIn = 100，currentBet = 100）
- Carol raise to 120：`120 >= 100` → raise 被阻止 ✅
- Carol raise to 80：`80 >= 100` → raise 被允许 ❌

Carol raise 80 会导致：

1. `activePlayer.bet` 从 0 变成 80
2. `currentBet` 变成 80
3. 桌面 pot 被添加 80（从 Carol 的 stack）
4. Bob 的 call 100 > 80，Bob 仍需补足差额

**但实际风险评估**：

- `minRaiseTo = currentBet + minBet`（line 323）会阻止 raise 到 80（除非 minBet = 0）
- 德州扑克规则中，raise 必须至少达到 currentBet + minBet
- 因此 80 < 100（currentBet）会被 `minRaiseTo` 约束阻止

**结论**：P2-NEW-001 的实际漏洞已被 `minRaiseTo` 约束缓解，但 `calledAllIn` 保护逻辑仍显冗余（允许 raise 到 calledAllIn 以下金额在 minRaiseTo 约束下不可能）。可保持现状或简化保护逻辑。

**状态**: 🟡 非严重，建议 P2 优化而非紧急修复
