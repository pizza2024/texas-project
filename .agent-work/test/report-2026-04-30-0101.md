## 本轮 CodeReview 重点

### 新 Commit 审查（`fd1c112` — WebSocket Push on Deposit）

**文件:** `apps/backend/src/deposit/deposit.service.ts`

**功能实现:** 充值确认后通过 `wsManager.emitToUser` 推送 `deposit_confirmed` 事件。

```typescript
// deposit.service.ts:474-489
try {
  const updated = await this.prisma.wallet.findUnique({...});
  this.wsManager.emitToUser(userId, 'deposit_confirmed', {
    txHash, amount: amount.toNumber(), chips: chips.toNumber(),
    newBalance, timestamp: new Date().toISOString(),
  });
} catch (err) {
  this.logger.warn(`Failed to emit deposit_confirmed WS event...`);
}
```

**审查结果:**

- ✅ WebSocket 推送在事务外执行（正确 — 避免阻塞 DB commit）
- ✅ 推送后查询最新余额（正确 — 获取已提交值）
- ✅ 错误被 try/catch 包裹（可接受 — WS 失败不影响充值核心逻辑）
- ⚠️ `wsManager.emitToUser` 本身在 server 未初始化时无异常（silent no-op），但 deposit 流程依赖它只用于通知，风险极低
- ⚠️ **Bonus mission + DepositBonus 创建不在 atomic tx 内** — 并发充值场景下可能重复触发

### P2-DEPOSIT-BONUS-TX 详解

首次充值 bonus 逻辑（lines 428-466）在 `await this.prisma.$transaction` 块**之外**执行。结构如下：

```
1. $transaction { depositRecord + wallet + user.balance + transaction }  ← atomic ✅
2. $transaction 外: user.findUnique → hasReceivedFirstDepositBonus 检查
3. $transaction 外: missionService.progressMission + depositBonus.create + user.update
```

并发场景：用户快速发起两笔充值（足够间隔内），`hasReceivedFirstDepositBonus` 两次都读到 `false`，可能导致 `progressMission` 被调用两次（mission service 自身应有幂等保护），但 `depositBonus.create` 会因 unique constraint 而失败（Prisma 报错外溢）。

**风险评估:** 中低 — 实际发生概率低（充值间隔需 < 30s），且 Prisma unique constraint 提供保护。
