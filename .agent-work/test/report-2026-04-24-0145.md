## 本轮 CodeReview 发现

### P2 — 新观察（4项）

#### 1. `clearTableState` 静默吞掉错误（P2-Low）

**文件**: `table-manager.service.ts` 第 237-243 行

```typescript
try {
  await this.prisma.table.deleteMany({ where: { id: roomId } });
} catch {
  await this.prisma.table
    .updateMany({ where: { id: roomId }, data: { stateSnapshot: null } })
    .catch(() => {}); // ← 二次静默吞掉
}
```

**问题**: 外层 catch 只记录了日志，内层 updateMany 的错误被 `.catch(() => {})` 完全吞掉。如果 P2003 外键约束失败后的 fallback 也失败（如 DB 连接问题），调用方无任何感知。
**风险**: 低 — 属于防御性清理逻辑，不会影响游戏核心流程。
**建议**: 内层 `.catch()` 至少应该记录 `logger.warn`。

---

#### 2. `hand-history.service.ts` JSON.parse 静默失败（P2-Low）

**文件**: `hand-history.service.ts` 第 100-102 行

```typescript
} catch {
  holeCards = ['??', '??'];
}
```

**问题**: 解析 `stateSnapshot` 失败时返回 `'??', '??'`，但调用方（API 响应）不会收到任何错误提示，用户看到的手牌是 `"??"` 而非"数据不可用"。
**风险**: 低 — 历史牌局记录损坏时用户体验降级，不影响游戏逻辑。
**建议**: 考虑在 API 响应中标记 `cardsRevealed: false` 而非静默替换为 `'??'`。

#### 4. `wallet.getBalance` 存在 dual-source 降级（P2-Low）

**文件**: `wallet.service.ts` 第 29-34 行

```typescript
const user = await this.prisma.user.findUnique({
  where: { id: userId },
  select: { coinBalance: true },
});
return user?.coinBalance ?? WalletService.STARTING_CHIPS;
```

**问题**: 当 `Wallet` 记录不存在时，fallback 到 `User.coinBalance`。这意味着同一用户可能同时存在 `Wallet.chips` 和 `User.coinBalance` 两个余额源，长期来看有数据不一致风险。
**风险**: 低 — 目前新用户注册时会自动创建 Wallet（`setBalance` upsert），但手动 DB 操作或遗留数据可能触发此路径。
**建议**: 确认产品意图是只用 `Wallet.chips` 还是 `coinBalance` 作为备用，并在代码中明确注释。
