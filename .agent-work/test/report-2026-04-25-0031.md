## P2 遗留问题审查

### P2-NEW-001 🟡 — calledAllIn 重入保护缺陷

**文件**: `table-engine/table-player-ops.ts` lines 315-318

**问题场景**：Alice all-in 100 → Bob call（calledAllIn=100, Bob.bet=100）→ Carol raise to 150

检查 'raise' case：

```typescript
if (
  this.table.calledAllIn !== null &&
  activePlayer.bet >= this.table.calledAllIn
) {
  return false; // blocks raise
}
```

Carol 的 `activePlayer.bet`（0） < `calledAllIn`（100）→ 条件为 false → **允许 raise**！

Alice all-in 100, Bob call, Carol 可以 raise to 150 — 这是规则漏洞。

**修复建议**: raise 保护条件应检查当前桌面的 `currentBet`（即 Bob 的 100），而非 `calledAllIn`。或添加标志位表示"all-in 已被跟注"后禁止任何 raise。

**状态**: 🟡 待确认是否为设计意图

### P2-NEW-002 🟡 — 牌局恢复时双重余额恢复

**文件**: `table-manager.service.ts` lines 119-155

恢复流程：

1. 从 Redis/SQLite 恢复 table snapshot（含玩家 stack）
2. 对每个玩家调用 `freezeBalance(userId, amount)`

**潜在问题**：如果玩家在崩溃前已在桌边（balance 已冻结），恢复时再次 `freezeBalance` 会再次 decrement `coinBalance`。但实际上 snapshot 的 stack 应该等于崩溃前的 frozen amount，所以不会 double-deduct。

**评估**: 需要实际测试验证。建议确认 snapshot 恢复的 stack 与冻结金额一致。

**状态**: 🟡 需测试验证
