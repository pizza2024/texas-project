### 🔴 P0 — 5个测试持续失败（app.gateway.spec.ts）

**文件**: `apps/backend/src/websocket/app.gateway.spec.ts`
**现状**: 6 passed, 5 failed / 11 total（较上轮9个失败有改善）

#### 根因分析

TimerService 从 AppGateway 拆分后，测试中 `scheduleActionTimeout` 的 mock 方式与新架构不兼容。具体5个失败用例：

| # | 用例 | 失败原因 |
|---|------|----------|
| 1 | `syncs the room state when a seated player disconnects` | `table.resetToWaiting` / `table.beginReadyCountdown` 未被调用（timer mock 未正确触发） |
| 2 | `runs settlement countdown, enters ready countdown, and then auto-starts` | 同上，Ready 倒计时 mock 未触发 |
| 3 | `rebuilds an in-progress settlement timer from restored state` | settlement timer 恢复逻辑 mock 问题 |
| 4 | `auto-checks on timeout when checking is allowed` | `table.beginActionCountdown` 未被调用 — `scheduleActionTimeout` 现在在 TimerService 而非直接挂在 gateway |
| 5 | `auto-folds on timeout when checking is not allowed` | `table.processAction` 未被调用 — 同上根因 |

**根因**: 测试中 `scheduleActionTimeout` 通过 `beforeEach` 注入的 mock 返回 `undefined`，但新架构中 `TimerService.scheduleActionTimeout()` 是真正执行 timer 逻辑的地方。测试 mock 需要改为对 `TimerService` 的 mock，而非直接对 gateway 方法的 mock。

**修复方向**: 测试文件需要更新以匹配新的 TimerService 架构——对 `TimerService.scheduleActionTimeout()` 进行 mock，而不是直接调用 `(gateway as any).scheduleActionTimeout()`。
