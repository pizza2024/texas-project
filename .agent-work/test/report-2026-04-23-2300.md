### 🟠 P1 — 重要问题 (2个) — **与上次报告相同，未修复**

#### 1. Withdraw 服务的 In-Memory Cooldown 在多实例部署下可被绕过

**文件**: `withdraw/withdraw.service.ts` (行 ~51)  
**问题**: `cooldowns` Map 作为 Redis 不可用时的 fallback，仅存储在单进程内存中。  
**影响**: 提现冷却机制在分布式环境下失效，用户可快速重复提现请求。  
**状态**: 未修复 — 建议 Coding 代理优先处理  
**建议**: 使用 Redis SETNX + TTL 实现分布式锁

#### 2. JWT Strategy 的 Redis 不可用时跳过 Session 验证

**文件**: `auth/jwt.strategy.ts` (行 22-28)  
**问题**: Redis 不可用时允许认证通过，单设备登录保护失效。  
**影响**: Redis 故障期间，Token 被盗后可无限制使用。  
**状态**: 未修复 — 已在 task-queue 中但未处理  
**建议**: 添加 `SECURITY` 级别日志和可选 strict mode

### 🟡 P2 — 优化建议 (4个)

#### 3. AppGateway 房间密码验证缺少暴力破解防护

**文件**: `websocket/game.handler.ts` (行 100-108)  
**描述**: `bcrypt.compare` 失败时 emit `wrong_password`，但无日志记录和速率限制。  
**建议**: 添加 WebSocket 层级的房间密码尝次计数，超限后临时封禁 IP。

#### 4. WebSocket Rate Limit 使用 In-Memory Map Fallback

**文件**: `websocket/app.gateway.ts` (行 127)  
**描述**: Redis 不可用时回退到 in-memory Map，无法跨进程共享。  
**建议**: 已在 4bccde4 commit 中优化，当前回退为有记录的安全 trade-off。

#### 5. 房间列表页组件过大（Productor P1 关联）

**文件**: `apps/web/app/rooms/page.tsx` — **1553 行**  
**确认**: 与 Productor 报告一致，此文件过大难以维护。  
**建议**: 拆分为 RoomCard、FilterBar、RoomList 等子组件。

#### 6. 移动端页面仍较大（758 行）
