### 🛑 Critical（新发现）

#### B1 — `advanceStreet` 在 `eligible.length === 0` 时缺少 `return` — NaN 进入 `winAmounts`

**文件:** `table-engine/table-round.ts`, `performShowdown()` (lines 155–244)

```typescript
if (eligible.length === 0) {
  this.table.currentStage = GameStage.SETTLEMENT;
  return; // ← 缺少！继续执行下面的 buildPots 导致 NaN
}
// 下面当 eligible=[] 时仍可到达：
const pots = this.buildPots(allPlayers); // eligible=[] 导致问题
const share = Math.floor(pot.amount / potWinners.length); // NaN 当 potWinners=[]
```

**影响:** 当所有玩家在 showdown 前弃牌时（如竞态条件导致 `advanceStreet` 在不应调用时调用），`winAmounts` 被污染为 `NaN`，`lastHandResult` 中每个玩家的 `winAmount` 都是 `NaN`，数据持久化到 DB 并广播给客户端。

**修复:** 在 `eligible.length === 0` 分支的 `this.table.currentStage = GameStage.SETTLEMENT;` 后添加 `return`。

**优先级:** P0（游戏数据损坏）

### 🔴 High（新发现）

#### B2 — `client.leave(roomId)` 在获取 room lock 之前调用

**文件:** `websocket/game.handler.ts`, `handleLeaveRoom()` (lines 365–411)

```typescript
return gateway.withUserLock(userId, async () => {
  return gateway.withRoomLock(roomId, async () => {
    client.leave(roomId);  // ← 先离开 Socket.io room
    const result = await gateway.tableManager.leaveCurrentRoom(userId); // ← DB 更新在后
  });
});
```

**影响:** `client.leave()` 在 room lock 获取之前和 DB 状态更新之前执行，导致竞态：game-state 广播可能发送到已离开的 room（玩家已离开视觉，但服务器仍广播）。

**修复:** 将 `client.leave(roomId)` 移至 `leaveCurrentRoom()` 之后。

**优先级:** P1（竞态条件，数据一致性）

#### B3 — `freezeBalance` 无 `availableBalance >= amount` 校验

**文件:** `wallet/wallet.service.ts`, `freezeBalance()` (lines 149–156)

```typescript
async freezeBalance(userId: string, amount: number): Promise<void> {
  const normalized = Math.max(0, amount);
  await this.prisma.$transaction([
    this.prisma.wallet.upsert({ ... }),   // 设置 frozenChips = normalized
    this.prisma.user.update({            // 扣减 coinBalance
      data: { coinBalance: { decrement: normalized } },
    }),
  ]);
  // 无 availableBalance >= amount 检查 — 如果 amount > chips，coinBalance 变为负数
}
```

**影响:** 如果 `amount > chips`，`coinBalance` 会变为负数。结合 B2 中的 balance 重放问题，理论上玩家余额可变为负数。

**优先级:** P1（资金数据完整性）

#### B4 — Bot prefix 碰撞导致 `coinBalance` 从不同步

**文件:** `wallet/wallet.service.ts`, `setBalances()` (lines 75–104)

```typescript
if (userId.startsWith(BOT_ID_PREFIX)) {
  return [walletOp]; // 跳过 coinBalance 同步
}
```

**影响:** `userId` 以 `BOT_ID_PREFIX` 开头的真实用户（如 `"bot_player123"`），其 `wallet.chips` 更新但 `user.coinBalance` 永不同步，导致 `getAvailableBalance` 返回错误的可用余额。

**修复:** 使用 `User.isBot` 标志位或检查 `User.role` 字段，而非字符串前缀匹配。

**优先级:** P1（数据一致性）

### 🟡 Medium（新发现）

#### B5 — `exchangeBalanceToChips` 的 `getRealBalance` 读取在事务外

**文件:** `wallet/wallet.service.ts`, `exchangeBalanceToChips()` (lines 329–379)

USDT 余额检查在 `$transaction` 外执行，两个并发请求可能同时通过检查后各自扣款，导致 USDT 超支（对比 `exchangeChipsToBalance` 正确地在事务内检查）。

#### B6 — ELO 边界强制执行与 ELO 更新是分离的事务

**文件:** `matchmaking/matchmaking.service.ts`, `updateElo()` (lines 220–243)

两次独立 `$transaction` 之间服务器崩溃，ELO 值可能短暂超出 [ELO_MIN, ELO_MAX] 范围，影响匹配配对。

#### B7 — `getBalance` 返回总额而非可用额，依赖调用方 discipline

**文件:** `wallet/wallet.service.ts`, `getBalance()` (lines 19–40)

返回 `wallet.chips`（总额）而非 `chips - frozenChips`（可用）。虽然关键金融路径正确使用 `getAvailableBalance`，但存在误用风险。

## Web CodeReview 发现

**审查文件:** `web-review.md` (本目录)

### 🟡 Medium（新发现）

#### W1 — WebSocket `socket.on` listeners 未在 `useEffect` cleanup 中移除

**文件:** `apps/web` — socket 相关 hooks

`socket.on` 注册的 `room_update`、`emoji-reaction`、`already_in_room`、`disconnect`、`room_full`、`insufficient_balance`、`wrong_password`、`error` 等 listeners 从不清理，每次 hot reload 导致 listener 累积。

#### W2 — `TableState` / `Player` 类型在前端和 `packages/shared` 之间重复定义且漂移

`consecutiveTimeouts` 字段在 shared types 中缺失但前端使用。类型定义两处维护，存在 drift 风险。

#### W3 — 无客户端操作验证；按钮可重复提交

ActionBar 的 fold/call/raise/allin 按钮无防重复提交机制，在网络延迟时玩家可能多次提交同一操作。

#### W4 — Token 过期时使用 `window.location.replace` 丢弃游戏状态

`handleExpiredSession` 使用 `window.location.replace`，玩家在游戏中 token 过期会丢失完整游戏状态，无法恢复。

#### W5 — 无 Refresh Token 机制

Token 过期期间的游戏会话无法续期，玩家被迫重新登录。

### 🔴 Critical（新发现）

#### W6 — 无 React.memo，game table 组件全部 re-render

每次 `room_update` 触发完整的 table re-render，级联到所有子组件，性能问题在多人游戏中加剧。
