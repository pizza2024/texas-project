## P0 — T-009: app.gateway.spec.ts 4个测试持续失败

**文件**: `apps/backend/src/websocket/app.gateway.spec.ts`
**测试结果**: 4 failed, 7 passed / 11 total

### 根因分析（已确认）

TimerService 重构后（commit `298dce2` W-006），`scheduleActionTimeout` 逻辑从 AppGateway 移至 TimerService。但 mock 架构未适配：

1. **TimerService mock** 在测试中仅存在于 `gateway.timerService`（通过 `providers: [TimerService, ...]` 注入），其 `finalizeActionTimeout` 等方法在 `jest.advanceTimersByTimeAsync(20000)` 触发时，**内部调用 `table.getTimeoutAction()` 和 `table.processAction()`，但 mock 的 `beginActionCountdown` 只设置 `actionEndsAt`，不触发真实的超时回调链**

2. **测试期望**在 `advanceTimersByTimeAsync(20000)` 后验证 `table.getTimeoutAction` 和 `table.processAction` 被调用，但 TimerService 的 `setTimeout` 回调中实际调用这些方法时，mock 的 table 对象并未被正确触发

### 4个失败用例

| 用例 | 失败原因 |
|------|---------|
| `runs settlement countdown, enters ready countdown, and then auto-starts` | `resetToWaiting` 未调用 — TimerService mock 的 `beginSettlementCountdown`/`beginReadyCountdown` 只记录调用，未推进游戏状态 |
| `rebuilds an in-progress settlement timer from restored state` | 同上 |
| `auto-checks on timeout when checking is allowed` | `table.getTimeoutAction` 未被调用 — `setTimeout` 回调未触发 |
| `auto-folds on timeout when checking is not allowed` | `table.processAction` 未被调用 |

### 修复方案

TimerService mock 需要在 `begin*Countdown` 中同步调用对应的 table 状态推进方法：

```typescript
// 在 mock 的 beginSettlementCountdown 中
beginSettlementCountdown: jest.fn((ms: number) => {
  table.currentStage = GameStage.SETTLEMENT;
  table.resetToWaiting?.(); // 触发状态重置
  table.beginReadyCountdown?.(ms);
}),

// 在 mock 的 beginActionCountdown 中（针对超时测试）
beginActionCountdown: jest.fn((ms: number) => {
  table.actionEndsAt = Date.now() + ms;
  // 当测试调用 scheduleActionTimeout 时，需模拟 setTimeout 回调
  // 在 setTimeout(fn, ms) 调用时立即执行 fn，而不是等待 advanceTimersByTimeAsync
  // 或在 mock 中存储回调并在 advanceTimersByTimeAsync 时触发
}),
```

**核心问题**: `scheduleActionTimeout` 使用 `setTimeout` 注册回调，但 `jest.advanceTimersByTimeAsync` 需要真正等待对应时间才能触发。mock 的 `beginActionCountdown` 设置了 `actionEndsAt` 但没有真正触发 TimerService 内部的超时逻辑。
