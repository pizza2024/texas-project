### 遗留 P1 问题深度分析

#### P1-NEW-004: `exchangeChipsToBalance` TOCTOU — **确认为 fund loss 漏洞**

```typescript
// Line 275 — 余额检查在事务外
const availableChips = await this.getAvailableBalance(userId);  // ← T1 读取: 1000
if (chipsAmount > availableChips) { ... }                       // ← T1 通过

// Line 285-305 — 事务内再次读取
const wallet = await tx.wallet.findUnique({ where: { userId } });
const currentBalance = wallet?.chips ?? ...;                    // ← T1 读取: 1000
const newBalance = Math.max(0, currentBalance - chipsAmount);   // = 0
```

**攻击路径：并发双花**

1. T1: `getAvailableBalance` = 1000, 通过检查
2. T2: `getAvailableBalance` = 1000, 通过检查（同一用户同时发起两笔提现）
3. T1: 事务内 `currentBalance` = 1000, 扣减后 newBalance = 0, 提现1000成功
4. T2: 事务内 `currentBalance` = 1000（读于T1提交前），扣减后 newBalance = 0, 提现1000成功
5. **结果：** 用户收到2笔USDT提现，但DB只扣了1笔chips — **fund loss**

**修复方向：** 余额检查应移至 `$transaction` 内部，基于 `currentBalance`（已在事务内读取）进行校验，不应独立调用 `getAvailableBalance`。

#### P1-NEW-003: `setBalances` Phase 2 非原子 — **数据一致性风险**

```typescript
// Phase 1 — 原子（已提交）
await this.prisma.$transaction(entries.map(...));  // ✅ 全部成功

// Phase 2 — 非原子，独立更新
const phase2Results = await Promise.allSettled(
  entries.map(({ userId, balance }) =>
    this.prisma.user.update({ where: { id: userId }, data: { coinBalance: normalized } })
  )
);
// 如果部分失败：Phase1已提交，Phase2部分成功 → wallet.chips ≠ user.coinBalance
```

**风险：** Phase 1 已提交，Phase 2 部分失败时，wallet 和 user.coinBalance 不一致。
当前有 rethrow 机制（P1-WALLET-002），但 Phase 1 的 wallet 更新无法回滚。

#### P1-NEW-010: `checkStaleProcessing` 无幂等保护 — 重复入队风险

```typescript
for (const request of stale) {
  if (!request.txHash) {
    await this.withdrawQueue.enqueueWithdraw(request.id); // 无幂等键
  }
}
```

同一 stale 请求在下次 cron 触发时可能重复入队。BullMQ 队列本身支持幂等，但这里没有传递 idempotency key。

### `resetToWaiting` 不清理 rakeAmount/rakePercent（P2-NEW-021）

```typescript
resetToWaiting(): void {
  // ...
  this.table.rakeAmount = 0;       // ← 没有
  this.table.rakePercent = 0;       // ← 没有

```
