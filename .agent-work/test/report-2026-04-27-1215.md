### P1 — New Issues

#### P1-BLAST-009 — `distributePrizes()` not atomic (Prize distribution can be partial)

**File:** `blast.service.ts:504–541`
**Severity:** P1 — Financial integrity
**Description:** `distributePrizes()` loops over rankings and calls `unfreezeAndAward()` + `prisma.transaction.create()` sequentially **without a wrapping DB transaction**. If the process crashes or an error occurs mid-loop, some players receive their prizes while others do not — leaving the game in an inconsistent state with no recovery path.

```ts
// Current (vulnerable):
for (let i = 0; i < rankings.length; i++) {
  await this.walletService.unfreezeAndAward(playerId, prizeChips);
  await this.prisma.transaction.create({ data: {...} }); // No atomicity across iterations
}
```

**Risk:** Blast game ends → 3 players ranked → crash on 2nd iteration → 1st player paid, 2nd and 3rd not paid. Real money at stake.
**Fix:** Wrap entire loop in `this.prisma.$transaction()` or batch all `unfreezeAndAward` calls.

#### P1-BLAST-011 — `drawMultiplier()` uses `Math.random()` for prize multiplier

**File:** `blast.service.ts:430`
**Severity:** P1 — Game fairness
**Description:** The Blast multiplier (which determines the prize pool size — 2x, 5x, or 10x) is drawn using `Math.random()`, which is not cryptographically secure. For a gambling game where the multiplier IS the core game outcome, this is a fairness concern. An attacker who can predict or bias the RNG could selectively join games with favorable multipliers.

```ts
private drawMultiplier(): number {
  const roll = Math.random() * 100; // NOT cryptographically secure
  if (roll < 60) return 2;
  ...
}
```

**Note:** The code itself acknowledges this is Phase 2 simplified: _"NOTE: This is a simplified version. The full version (Phase 3) will use `drawBlastMultiplier()` from @texas/shared/types/tournament."_ Phase 3 must use `crypto.randomInt()` or equivalent.
**Fix:** Replace with `crypto.randomInt(1, 100)` in Phase 3.

### P2 — Recommendations

#### P2-BLAST-013 — `activeGames` Map is process-local (singleton memory)

**File:** `blast.service.ts:71`
**Severity:** P2 — Scalability
**Description:** `activeGames` is a plain `Map<string, BlastGameRecord>` in `BlastService` (singleton). In a multi-instance NestJS deployment (e.g., Kubernetes replicas), each instance has its own `activeGames` copy. If the instance handling `startBlastGame` differs from the one handling `endBlastGame` (via setTimeout fired on same instance), or if a player reconnects to a different instance, game state may be lost.
**Partial mitigation:** `persistGameRecord()` writes to Redis (line 197), but in-memory Map is still consulted first (line 194). Recovery on instance restart relies on Redis, but mid-game instance switches are not handled.
**Fix:** Use Redis as primary store for active game records, or implement sticky sessions at the load balancer.

#### P2-BLAST-014 — `setTimeout` for game timer not clustered-aware

**File:** `blast.service.ts:209–213`
**Severity:** P2 — Reliability

```ts
setTimeout(() => {
  this.endBlastGame(lobbyId).catch(...)
}, delayMs);
```

If the NestJS process restarts before the timeout fires, the scheduled `endBlastGame` is lost. Redis persistence exists (`persistGameRecord`) but the timeout itself is not recovered. For a 3-minute game, this means if the server restarts at minute 2, the game may never end automatically.
**Fix:** Use a persistent job queue (BullMQ with delayed job) instead of `setTimeout` for game timers.

#### P2-BLAST-015 — `forfeitBlast` playerIds mutation while iterating

**File:** `blast.service.ts:371–417`
**Severity:** P2 — Logic bug risk
**Description:** `forfeitBlast` mutates `game.playerIds` (filters out the departing player at line 381) and then emits to `game.playerIds` (now the filtered list). This is correct, but the mutation of the shared `BlastGameRecord` object is a side effect. If multiple players forfeit concurrently, or if `onBlastHandComplete` is called during this mutation, state could be inconsistent.

```ts
game.playerIds = game.playerIds.filter(id => id !== playerId); // Mutates shared record
// ...
for (const remainingId of game.playerIds) { // Uses mutated array
```

**Fix:** Use immutable pattern — create a new array rather than mutating.
