### 🔍 P2-NEW-001 重新审查 — calledAllIn 重入保护

**文件**: `table-engine/table-player-ops.ts` lines 316-322

**上次报告描述的场景**：Alice all-in 100 → Bob call（calledAllIn=100）→ Carol raise to 150

**raise case 当前代码**（line 316-322）:
```typescript
case 'raise': {
  if (
    this.table.calledAllIn !== null &&
    amount >= this.table.calledAllIn
  ) {
    return false;
  }
```

**关键区别**：上次报告写的是 `activePlayer.bet >= calledAllIn`，但实际代码是 `amount >= calledAllIn`（比较的是 raise 的目标金额，而非当前玩家已下注）。

**分析**：
- Alice all-in 100，calledAllIn = 100
- Bob call（calledAllIn 已是 100）
- Carol raise to 150：检查 `150 >= 100` → **条件为 true → raise 被阻止**
- Carol raise to 80：检查 `80 >= 100` → 条件为 false → **raise 被允许**

**问题**：Carol 可以在 Alice all-in 100 后 raise to 120（超过 Alice 的 100 但低于桌面 currentBet + minRaise）。这是规则漏洞。

**但需注意**：桌面 currentBet 在 Bob call 后已是 100，minRaise 约束（line 323 `minRaiseTo = currentBet + minBet`）理论上应阻止低于 currentBet 的 raise。但 Carol 的 `activePlayer.bet`（0）< `currentBet`（100），而她仍可用 120 超过 Alice 的 100 并重新加注。

**建议修复**：保护条件应检查 raise 后 total bet 是否超过桌面 currentBet，而非单纯比较 calledAllIn。当前 `minRaiseTo` 逻辑本身应足够，但仍存在边界 case 风险。

**状态**: 🟡 规则漏洞存在，建议 P2 修复

### ⚠️ P2-NEW-004 — roomId 未从 payload 验证

**文件**: `game.handler.ts` lines 240-261

```typescript
data: { action: unknown; amount?: unknown; roomId?: unknown },
```

`roomId` 在 schema 中为 optional（`z.string().min(1).optional()`），handler 从两个来源获取：
1. `validated.roomId`（可选，用户可控）
2. `gateway.tableManager.getUserCurrentRoomId(userId)`（fallback）

如果用户提供了 `roomId`，代码直接使用而非验证归属。攻击者可能借此向其他房间发送操作。

**修复建议**：
- 从 `@ConnectedSocket` 的 `socket.data.roomId` 获取（已由 `@WebSocketSubscribe` 验证）
- 如果用户提供了 `roomId`，应与 `socket.data.roomId` 交叉验证

**状态**: 🟡 待修复
