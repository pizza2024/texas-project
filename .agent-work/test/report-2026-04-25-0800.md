**总计: 19 个 `error TS`，全部在 spec / e2e 文件**

| 文件                               | 错误数 | 类型                                           |

**影响:** `npm run test` 仍然全部通过（267/267），Jest 运行时忽略类型错误。但 `npx tsc --noEmit` 失败。

#### connection-state.service.spec.ts — 12 errors ⚠️

```ts
// 问题: redis.incr 类型签名与 mock 不匹配
redis.incr!.mockResolvedValue(3); // TS2339: Property 'mockResolvedValue' does not exist on type '(key: string, ttlSeconds?: number) => Promise<number | null>'

// 原因: incr 的类型声明不支持 Jest mock
// 修复方向: 将 redisService mock 对象的 incr 属性类型断言为 jest.Mock
// 示例: (redis.incr as jest.Mock).mockResolvedValue(3)
// 或在 describe 块顶部: const incrMock = redis.incr as jest.Mock;
```

#### rakeback.controller.spec.ts — 5 errors ⚠️

```ts
// 问题: as Request 双重断言缺失
const req = { user: mockUser } as Request; // TS2352

// 修复: 使用双重断言
const req = { user: mockUser } as unknown as Request;
// 或更好的方式: 在测试文件顶部添加类型缩小
```

#### rakeback.e2e-spec.ts — 2 errors ⚠️

```ts
// 问题: Prisma create 使用 passwordHash 字段
passwordHash: '$2b$12$...',  // TS2561: Did you mean 'password'?

// 修复: 将 passwordHash 改为 password
// Prisma User model 使用 password 字段，passwordHash 是内部存储实现
```
