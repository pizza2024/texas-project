#### 3. P2-CHAT-STUB — `handleClaim` 无实际 API 调用

**文件:** `apps/web/app/missions/page.tsx:294`
**问题:** `handleClaim` 只做 `setTimeout(500ms) + GET /missions` 刷新，注释称"backend auto-claims"。这是 **正确设计** — 任务完成时 backend 自动发放奖励，前端只负责刷新 UI。
**修复方向:** 确认这是预期行为后可关闭，无需修改
**风险:** 低 — 架构设计而非缺陷

#### 4. P2-ROUTER-ANY — `router.events` 使用 `as any`

**文件:** `apps/web/app/rooms/page.tsx:937`
**问题:** `(router as any).events?.on("routeChangeComplete", handleRouteChange)` — Next.js 13+ App Router 中 `router.events` 仅在 Pages Router 可用，需类型断言。
**修复方向:** 确认是否使用 App Router；如是，考虑移除该监听或使用 `usePathname` 替代
**风险:** 低 — 运行时问题概率低

#### 5. P2-ROOM-RETRY — 重试逻辑无指数退避

**文件:** `apps/web/app/deposit/page.tsx:107`
**问题:** `setTimeout(() => loadData(true), 800)` 固定 800ms 重试，无指数退避。
**修复方向:** 实现 `Math.min(800 * 2 ** retry, 30000)` 指数退避
**风险:** 低 — 仅影响失败场景的 UX

#### 6. P2-WS-RATE-UNIT — PASSWORD_ATTEMPT_WINDOW_MS 单位混淆

**文件:** `apps/backend/src/websocket/connection-state.service.ts:25`
**问题:** 代码中显示 `PASSWORD_ATTEMPT_WINDOW_MS=*** * 60 * 1000`，实际值为 `5 * 60 * 1000 = 300000ms`。注释称"5 minutes"，但 `windowSec` 变量计算正确。注释中的 `***` 疑似日志过滤遮蔽，非实际代码问题。
**修复方向:** 确认 `PASSWORD_BAN_MS` 与 `PASSWORD_ATTEMPT_WINDOW_MS` 关系
**风险:** 低 — 代码逻辑正确

#### 7. P2-CODE-PATTERN — mission.service `Promise.all` 无错误处理

**文件:** `apps/backend/src/mission/mission.service.ts:170`
**问题:** `Promise.all(MISSION_DEFINITIONS.map(...))` — 任一 upsert 失败会导致全部回滚。无 `Promise.allSettled`。
**修复方向:** 改为 `Promise.allSettled` 并汇总失败项单独 log
**风险:** 低 — mission seed 在 OnModuleInit 中，影响启动而非运行时

#### 8. P2-CHAT-IDEMPOTENCY — 聊天消息无幂等键

**文件:** `apps/backend/src/websocket/game.handler.ts`
**问题:** 聊天消息 broadcast 后无幂等键，前端重发可能导致重复。
**修复方向:** 添加 `nonce` 或 `idempotencyKey` 字段去重
**风险:** 低 — 罕见网络重试场景
