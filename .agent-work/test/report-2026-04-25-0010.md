### P0 — 紧急（安全/资金风险）

#### P0-001 — 🔴 Auth 模块：Redis 宕机时 JWT 验证被完全绕过
**文件**: `auth/jwt.strategy.ts` lines 22–42
**严重性**: CRITICAL — 安全认证可被绕过

Redis 不可用时，`jwt.strategy.ts` 记录 `[SECURITY-AUTH-BYPASS]` 但**仍然返回有效用户**，导致单设备登录保护完全失效。攻击者只需在 Redis 故障期间持有有效 JWT，即可无限制使用。

```typescript
if (!this.redisService.isAvailable) {
  this.logger.error(`[SECURITY-AUTH-BYPASS] ...`);
  // ← Falls through, 仍返回有效用户
}
return { userId: payload.sub, username: payload.username, role: payload.role ?? 'PLAYER' };
```

**影响**: 任何失窃 JWT 在 Redis 宕机期间均可无限使用。
**修复**: Redis 不可用时应 `throw UnauthorizedException`，或维护数据库级 Denylist。

---

#### P0-002 — 🔴 Withdraw 模块：Redis 不可用时冷却时间被静默跳过
**文件**: `withdraw/withdraw.service.ts` lines 97–130
**严重性**: CRITICAL — 可绕过冷却时间提取资金

`checkAndConsumeCooldown` 在 Redis 返回 `null` 时**直接 return**，调用方 `createWithdraw` 从不报错，导致冷却时间被静默绕过。

```typescript
private async checkAndConsumeCooldown(userId: string): Promise<void> {
  const acquired = await this.redisService.setNX(lockKey, ..., ttlSeconds);
  if (acquired === true) { this.cooldowns.delete(userId); return; }
  if (acquired === false) { throw BadRequestException; }
  // acquired === null (Redis 不可用) → return without throwing
  return;
}
```

**影响**: 分布式部署中 Redis 不可达时**所有用户均可无冷却提取资金**。
**修复**: Redis 不可用时应阻止提现而非静默通过。

#### P0-003 — 🔴 Withdraw 模块：processWithdraw 服务层无 Admin 角色验证
**文件**: `withdraw/withdraw.service.ts` lines 344–428
**严重性**: CRITICAL — 服务层鉴权缺失

`processWithdraw` 方法接收 `adminId` 参数但**从未验证调用者是否为 Admin**。Controller 层有 `AdminGuard`，但服务层本身是安全缺口——任何能调用该服务的代码均可执行管理员操作。

**影响**: 服务层鉴权缺失，违背纵深防御原则。
**修复**: 在服务层添加角色检查或依赖注入时确保只有 Guard 验证过的调用方可进入。

### P1 — 重要（功能/逻辑缺陷）

#### P1-001 — 🟠 Wallet 模块：freezeBalance/unfreezeBalance 不同步 User.coinBalance
**文件**: `wallet/wallet.service.ts` lines 133–175
**严重性**: HIGH — 数据一致性风险

`freezeBalance` 和 `unfreezeBalance` 只写 `Wallet` 表，从不动 `User.coinBalance`。而 `setBalance` 会同时写两者，导致冻结/解冻周期后 `User.coinBalance` 变得不准确。

**影响**: 任何读取 `User.coinBalance` 的代码（管理后台、getBalance 回退）会看到错误值。`rejectWithdrawRequest` 中的 `setBalance` 会用错误值覆盖。
**修复**: freeze/unfreeze 操作需同时更新 `User.coinBalance`。

---

#### P1-002 — 🟠 Auth 模块：verifyEmailCode 无速率限制（OTP 暴力破解）
**文件**: `auth/auth.controller.ts` lines 68–77
**严重性**: HIGH — 暴力破解风险

`requestEmailCode` 有 RateLimit（3次/小时），但 `verifyEmailCode` **没有任何速率限制**。攻击者可对 6 位数字 OTP（100万种可能）进行暴力枚举。

**影响**: 邮箱验证码可被暴力破解，导致账户被接管。
**修复**: 对 `verifyEmailCode` 添加基于 email 的速率限制（如 10 次/5 分钟）。
