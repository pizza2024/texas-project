## 🔴 P0 — 新发现（Critical Bug）

### P0-BRUTE-001: `checkPasswordAttemptLimit` 改为 async 后调用处未 await

**文件**: `apps/backend/src/websocket/game.handler.ts:99-104`

**问题描述**:
Commit `3e9ee55` 将 `checkPasswordAttemptLimit` 从同步改为 `async` 函数（返回 `Promise<'banned' | 'rate_limited' | 'ok'>`），但 `game.handler.ts` 中的调用处**未使用 `await`**：

```typescript
// game.handler.ts:99-104
const bruteForceResult = gateway.checkPasswordAttemptLimit(
  ipHash,
  roomId,
);                                                    // ← 返回 Promise，从不 "=== 'banned'"
if (bruteForceResult === 'banned') {                  // ← 永远是 truthy（Promise 对象）
  client.emit('rate_limited', { ... });
  return { event: 'rate_limited', data: { roomId } };
}
```

**影响**: 密码暴力破解保护**完全失效** — 每次调用都返回一个 Promise 对象，`=== 'banned'` 永远为 `false`，即使 IP+roomId 已达次数限制并被 Redis 标记为 banned，请求仍会继续进行密码验证流程。

**对比**：`checkRateLimit` 调用处（`game.handler.ts:46`）正确使用了 `await`：

```typescript
if (!(await gateway.checkRateLimit(userId))) { ... }  // ✅ 正确
```

**修复方案**:

```typescript
// game.handler.ts:99
const bruteForceResult = await gateway.checkPasswordAttemptLimit(
  ipHash,
  roomId,
);
```

## P1 — 待办问题

### P1-CLEAR-001: `clearPasswordAttempts` 调用未 await

**文件**: `game.handler.ts:123`

```typescript
gateway.clearPasswordAttempts(ipHash, roomId); // ← fire-and-forget
```

**影响**: 成功加入房间后清除 brute-force 计数器的操作可能未完成就退出了。由于 `clearPasswordAttempts` 内部是 Redis `del`，失败时静默忽略，属于边缘情况，**优先级低于 P0-BRUTE-001**。

**修复**: 改为 `await gateway.clearPasswordAttempts(ipHash, roomId);` 或显式 fire-and-forget（加注释说明）。

### P1-MOCK-001: `app.gateway.spec.ts` mock 类型不匹配

**文件**: `apps/backend/src/websocket/app.gateway.spec.ts`

**描述**: `checkPasswordAttemptLimit` 在 mock 中仍为同步 `jest.fn()`：

```typescript
checkPasswordAttemptLimit: jest.fn(),  // ← 应为 jest.fn().mockResolvedValue(...)
```

但该方法现已变为 `async`。由于 `jest.fn()` 在被 `await` 时会解析为 `undefined`（falsy），实际调用时会静默通过 brute-force 检查（表现为保护仍然部分有效，但行为不确定）。
