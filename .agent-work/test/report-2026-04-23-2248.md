### 🟠 P1 — 重要问题 (2个)

#### 1. Withdraw 服务的 In-Memory Cooldown 在多实例部署下可被绕过

**文件**: `withdraw/withdraw.service.ts` (行 51)  
**描述**: `cooldowns` Map 作为 Redis 不可用时的 fallback，仅存储在单进程内存中。在 K8s/Horizontal scaling 多实例部署时，用户可绕过 60 秒提现冷却。  
**影响**: 提现冷却机制在分布式环境下失效，用户可快速重复提现请求。  
**建议**: 使用 Redis 的 SETNX + TTL 实现分布式锁，或在数据库层面强制约束。

#### 2. JWT Strategy 的 Redis 不可用时跳过 Session 验证

**文件**: `auth/jwt.strategy.ts` (行 22-28)  
**描述**: Redis 不可用时，`validate()` 仅记录 WARN 日志后允许认证通过。代码注释说明这是有意的 trade-off 以避免服务完全中断。  
**影响**: Redis 故障期间，单设备登录保护失效，攻击者可能窃取 Token 后无限制使用。  
**建议**:

- 添加指标监控 `SECURITY` 级别日志
- 考虑可选的 "strict mode" 配置，在生产环境不允许降级
- 记录审计日志用于事后分析

### 🟡 P2 — 优化建议 (4个)

#### 3. AppGateway 房间密码验证缺少错误处理边界

**文件**: `websocket/game.handler.ts` (行 100-108)  
**描述**: `bcrypt.compare` 失败时 emit `wrong_password` 但没有记录安全审计日志。暴力破解房间密码无告警。  
**建议**: 添加 rate-limit + 日志记录。

#### 4. WebSocket Rate Limit 使用 In-Memory Map Fallback

**文件**: `websocket/app.gateway.ts` (行 127)  
**描述**: Redis 不可用时回退到 in-memory Map，无法跨进程共享。攻击者可从多个 Pod 绕过限制。  
**建议**: 考虑使用 Token bucket 算法作为 secondary 防护。

#### 5. 未发现 Table-Engine 核心扑克逻辑的明显 Bug

**检查结果**: `table-game-logic.ts` 使用 Fisher-Yates 洗牌算法，`hand-evaluator.spec.ts` 存在但需确认测试覆盖率。

#### 6. 代码组织良好，Handler 分离合理

**正面**: `game.handler.ts` 独立处理 join_room/player_action/leave_room/quick_match，职责清晰。
