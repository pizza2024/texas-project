## Problem

The e2e test for the Rakeback feature was failing due to:
1. **Circular dependency** when loading `AppModule` directly: `TableEngineModule → DepositModule → WebsocketModule`
2. **`$transaction` mock closure bug**: transaction mock created a fresh mock each call, preventing state from persisting between `findUnique` and `update`
3. **`mockResolvedValue` vs `mockImplementation` bug**: `mockResolvedValue` captures values at creation time, so `findUnique` always returned the initial balance even after `update` changed it
4. **Test expectations mismatch**: Tests expected 4 tiers but service has 5 (PLATINUM, DIAMOND added)

## Changes Made

### `apps/backend/test/rakeback.e2e-spec.ts` — Full rewrite

**Key fixes:**

1. **Avoid circular dependency**: Instead of loading `AppModule`, mock `RakebackController`, `RakebackService`, `WalletService`, `PrismaService`, and `JwtService` directly with `Test.createTestingModule()`

2. **`$transaction` mock passes same mock object**: Changed from `cb(buildMockPrisma(state))` (fresh mock each call) to `cb(mock)` (same mock reference)
   ```typescript

   ```

3. **`findUnique` uses `mockImplementation` not `mockResolvedValue`**: `mockResolvedValue` captures object values at creation time. `mockImplementation(() => ...)` reads live state.
   ```typescript
   // Before (BUG - captures values at creation)
   findUnique: jest.fn().mockResolvedValue({
     totalRake: state.totalRake,
     rakebackBalance: state.rakebackBalance,
   }),
   
   // After (CORRECT - reads live state)
   findUnique: jest.fn().mockImplementation(async () => ({
     totalRake: state.totalRake,
     rakebackBalance: state.rakebackBalance,
   })),
   ```

## Root Cause Analysis

1. **Circular dependency**: The `AppModule` imports modules that form a cycle. For e2e testing, mocking at the controller/service level is cleaner and avoids this.

2. **`$transaction` bug**: `buildMockPrisma(state)` creates a **new** mock object on each call to `$transaction`. When the service calls `user.update` inside the transaction, it writes to this new mock object. But `findUnique` (called outside the transaction in `claimRakeback`) reads from the original `mockPrisma`. Solution: create `mock` variable first, have `$transaction` call `cb(mock)` with the same reference.

3. **`mockResolvedValue` bug**: Jest's `mockResolvedValue(obj)` stores the `obj` reference. Since the object literal `{ totalRake: state.totalRake, ... }` is created ONCE at mock definition time, subsequent calls to `findUnique` return the same captured values. Solution: `mockImplementation(() => ({ ... }))` re-evaluates on each call.
