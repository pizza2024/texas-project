### P0 财务原子性缺陷（withdraw/deposit）

#### ✅ P0-Withdraw-001: executeChainWithdraw txHash 双花风险
- **文件**: `apps/backend/src/withdraw/withdraw.service.ts`
- **改动**: lines 530-555
- **原问题**: `txHash` 在 `tx.wait(1)` 确认前就保存到 DB，若 tx 失败/回滚，DB 已有 hash
- **修复**: 整体流程改为 1) `transfer()` 2) `tx.wait(1)` 3) 原子更新 `{ txHash, fromAddress, status: 'CONFIRMED' }`

#### ✅ P0-Withdraw-002: processWithdraw 幂等保护
- **文件**: `apps/backend/src/withdraw/withdraw.service.ts`
- **改动**: lines 394-407
- **原问题**: 无幂等保护，网络重试可导致双重链上转账
- **修复**:
  1. `executeChainWithdraw` 入口增加 `if (status === 'CONFIRMED') return txHash` 早返
  2. `processWithdraw` APPROVE 逻辑包入 `$transaction` 并内部重检 status，防止并发竞态

#### ✅ P0-Deposit-001: checkAddressDeposits 非原子操作
- **文件**: `apps/backend/src/deposit/deposit.service.ts`
- **改动**: lines 257-343
- **原问题**: 每个 event 的 `setBalance → depositRecord.create → transaction.create` 非原子，crash 可导致只扣余额无记录（重复充值）
- **修复**: 整个事件处理循环体包入 `this.prisma.$transaction(async (tx) => { ... })`，所有 DB 操作（wallet.upsert、user.update、depositRecord.create、transaction.create）均使用同一个 `tx` 回调客户端
