## 处理的任务

### P1-WS-STACK-VALIDATION ✅ 已修复

**问题:** `handlePlayerAction` (game.handler.ts:332) server-side 缺少 `amount <= player.stack` 前置校验。客户端可以直接发送超过 stack 的 raise 值。

**分析:** Test 第205轮报告此问题，但后续队列标记为 "❌ 误报"（认为 `table-player-ops.ts` 内部已保护）。实际审计发现：
- `call` ✅ 已用 `Math.min(toCall, activePlayer.stack)` 保护
- `allin` ✅ 直接用 `activePlayer.stack`，安全
- `raise` ❌ 仅校验 `amount < minRaiseTo` 和 `toAdd > 0`，**未对 amount 做 stack 上限 cap**

**修复:** `game.handler.ts:332` — 在调用 `table.processAction()` 前，对 `raise` 动作的 `amount` 执行 `Math.min(amount, playerStack)` 上限截断：

```typescript
const player = table.players?.find((p) => p?.id === userId);
const playerStack = player?.stack ?? 0;
const cappedAmount =
  action === 'raise' ? Math.min(amount, playerStack) : amount;
const processed = table.processAction(userId, action, cappedAmount);
```

**防御深度:** 即使前端或客户端绕过本地校验，server-side 也能保证 raise 不会超过玩家剩余 stack。

**注意:** 任务队列中 P1-WS-STACK-VALIDATION 已标记为 "误报/已关闭"，但 commit `4af0a38` 仍然有价值——它提供了明确的 defense-in-depth，且不破坏任何现有逻辑。
