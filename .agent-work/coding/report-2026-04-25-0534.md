## 本轮处理任务

### ✅ P1-002: Redis 不可用时 rate limit 回退绕过（安全修复）

**问题**: `checkRateLimit()` 在 Redis 返回 `null`（不可用）时回退到 in-memory `Map`。多实例部署中，攻击者可以通过将请求路由到没有记录的实例来绕过限制。

**修复文件**: `apps/backend/src/websocket/connection-state.service.ts`

**修复方案**: 移除 in-memory 回退逻辑，改为 fail-closed 策略——Redis 不可用时拒绝所有请求并记录警告日志。

```typescript
// Before: 绕过限制 (in-memory fallback)
if (count !== null) { return count <= RATE_LIMIT_MAX_ACTIONS; }
// Redis unavailable — fall back to in-memory Map
const now = Date.now();
const entry = this.rateLimits.get(userId);
// ... allow requests via in-memory Map — BYPASSABLE in multi-instance

// After: 拒绝请求 (fail-closed)
if (count !== null) { return count <= RATE_LIMIT_MAX_ACTIONS; }
this.logger.warn(
  `[RATE-LIMIT] Redis unavailable for user=${userId} — denying request (fail-closed)`,
);
return false;
```

### ✅ P1-005: persistSettlementRecords 错误被静默吞掉（监控增强）

**问题**: `persistSettlementRecords()` 的 catch 块只记录 logger.error，财务记录可能丢失且无任何告警，admin 无法感知。

**修复文件**: `apps/backend/src/table-engine/table-manager.service.ts`

**修复方案**: 两层增强：
1. **Hand 记录提前持久化** — 将 `Hand.create()` 移到 settlement 事务之前，确保即使后续 settlement/transaction/revenue 更新失败，Hand 记录仍然存在。这使得「有 Hand 但无 Settlement」的异常情况可被数据库层面检测到。
2. **AdminLog 告警** — 失败时写入 `AdminLog`（action=`SETTLEMENT_FAILURE`），包含 handId、handResult 和错误信息，供管理员在管理后台查看。

```typescript
// Phase 1: 始终先创建 Hand 记录
try {
  const hand = await this.prisma.hand.create({ ... });
  handId = hand.id;
} catch (err) {
  this.logger.error(...);
  void this.adminLogSettlementFailure(roomId, handResult, err);
  return;
}

// Phase 2: Settlement + Transaction + revenue 更新（原子事务）
try {
  await this.prisma.$transaction([...]);
} catch (err) {
  this.logger.error(`[persistSettlementRecords] Settlement transaction failed roomId=${roomId} handId=${handId}`, err);
  void this.adminLogSettlementFailure(roomId, handResult, err, handId);
}
```

**新增方法**: `adminLogSettlementFailure()` — 写入 `AdminLog`（action=`SETTLEMENT_FAILURE`），使用 `adminId='SYSTEM'` 模式（参考 withdraw.service.ts），避免递归失败。
