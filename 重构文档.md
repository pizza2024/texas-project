# 重构文档

> 基于"USDT充值积分设计"参考对话，对照现有系统，梳理差距并制定重构方向。

---

## 一、现状总览

### 当前技术栈

| 层次 | 技术 |
|------|------|
| 前端 | Next.js + Tailwind + shadcn/ui |
| 后端 | NestJS + Socket.IO |
| 游戏引擎 | TypeScript 状态机（单文件 table.ts） |
| 数据库 | SQLite + Prisma |
| 缓存 | ❌ 无 Redis |
| 队列 | ❌ 无 BullMQ |
| 支付 | ❌ 无 USDT / 链上支付 |

### 当前资产模型

```
User.coinBalance（唯一余额）
       ↓ join_room 时
Player.stack（内存）
       ↓ 结算倒计时结束
持久化写回 User.coinBalance + Wallet.balance
```

**当前只有一层资产，没有 Balance / Chips 分层。**

## 二、与参考架构的差距对比

### 2.1 资产体系（高优先级）

| 项目 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| 链上充值（USDT） | USDT → Balance | ❌ 未实现 | 缺整个充值链路 |
| 账户余额层 | `Balance`（可提现） | `coinBalance`（等同 Chips） | 无独立 Balance 层 |
| 游戏筹码层 | `Chips`（只能游戏用） | 直接用余额作为筹码 | 无 Chips 独立层 |
| 冻结余额 | `frozenBalance`（坐桌时锁定） | 字段存在但**从未使用** | 双开风险（同一余额进两桌） |
| 兑换流程 | Balance → Chips → Balance | ❌ 无兑换概念 | 缺整个兑换模块 |
| 提现流程 | Chips → Balance → USDT | ❌ 未实现 | 缺提现链路 |

**核心风险：** 当前系统用户可以在余额未冻结的情况下同时加入多个房间，导致超额使用余额。

### 2.2 Poker Engine 架构（中优先级）

| 模块 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| PlayerManager | 独立模块 | 内联在 `table.ts` | 可接受，但耦合高 |
| DeckManager | Fisher-Yates shuffle | ✅ 已实现 | 基本满足 |
| BettingManager | 独立下注模块 | 内联在 `table.ts` | 耦合高 |
| PotManager | 支持 Main Pot + Side Pot | ⚠️ **Side Pot 未实现** | All-in 场景结算错误 |
| HandEvaluator | 7选5 最优牌型 | ✅ 已实现 | 基本满足 |
| SettlementEngine | 独立结算模块 | 内联在 `table.ts` | 耦合高 |

**核心 Bug：** 当前 All-in 场景**未实现 Side Pot**，多人 All-in 时底池分配会出错（资金凭空消失或错误归属）。

### 2.3 游戏状态存储（高优先级）

| 项目 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| 桌子状态 | Redis | 纯内存 Map | 服务重启丢失所有游戏进度 |
| 状态持久化 | 关键事件写 Redis + PostgreSQL | 仅结算时写 SQLite | 中间手牌无备份 |
| 多实例扩展 | Redis Pub/Sub + 分片 | ❌ 无法水平扩展 | 单点故障 |

**核心风险：** 服务崩溃时，当前手牌的所有筹码变化全部丢失，资金无法恢复。

### 2.4 Tick Loop（中优先级）

| 项目 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| 操作处理 | ActionQueue + 统一 Tick | 事件驱动（直接执行） | 并发操作可能竞争 |
| 倒计时检查 | Tick 内统一检查 | `setInterval` 独立计时器 | 多个独立定时器，难以管理 |
| 广播时机 | 每 Tick 广播变化 | 每次操作后广播 | 基本等价，但缺乏节流 |

### 2.5 事务记录（高优先级）

| 类型 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| DEPOSIT | ✅ 记录 | ✅ 仅管理员操作时记录 | 无充值入口 |
| WITHDRAW | ✅ 记录 | ✅ 仅管理员操作时记录 | 无提现入口 |
| GAME_WIN | ✅ 记录 | ❌ 从不记录 | 无游戏盈亏历史 |
| GAME_LOSE | ✅ 记录 | ❌ 从不记录 | 无游戏盈亏历史 |
| EXCHANGE_IN | ✅ 记录 | ❌ 无兑换流程 | 缺整个模块 |
| EXCHANGE_OUT | ✅ 记录 | ❌ 无兑换流程 | 缺整个模块 |
| Settlement 记录 | ✅ 落库 | ❌ Schema 定义了但**从不写入** | 无单手盈亏记录 |

### 2.6 防洗钱 / 风控（高优先级 — 生产必须）

| 项目 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|
| 同桌频率限制 | 同一对玩家 24h ≤ 3 次 | ❌ 无限制 | 可任意对冲 |
| 对冲检测 | 统计 A→B 输率 | ❌ 未实现 | |

| 强制抽水 | 2%–8% | ❌ **完全无抽水** | 平台无收入，洗钱零成本 |

**核心风险：** 当前系统无任何抽水机制，洗钱成本为零，属于生产级**高危缺陷**。

### 2.7 匹配系统（低优先级）

| 项目 | 参考架构 | 现状 | 差距 |
|------|---------|------|------|

| 技能评级 | ELO / Glicko | ❌ 无 | |

### 2.8 发牌公平性（低优先级）

| 项目 | 参考架构 | 现状 | 差距 |

## 三、重构优先级路线图

### 🔴 P0 — 生产前必须解决（资金安全）

#### 1. 实现 frozenBalance（坐桌锁定）

**问题：** 玩家加入房间时余额不冻结，可双开同时消耗同一笔余额。

**改造方案：**
```
join_room 时：
  frozenBalance += buyIn
  availableBalance = balance - frozenBalance
  
leave_room / settlement 时：
  frozenBalance -= 原始buyIn
  balance = stack（游戏结果）
```

**涉及文件：**
- `apps/backend/src/wallet/wallet.service.ts` — 增加冻结/解冻方法
- `apps/backend/src/websocket/app.gateway.ts` — join_room 时调用冻结

#### 6. 构建三层资产体系（Balance / Chips 分离）

**问题：** 当前余额与筹码概念混用，无法支持未来充值提现。

**改造方案：**

```
当前：User.coinBalance（唯一）

目标：
  User.balance（法币/USDT等值，可提现）
  User.chips（游戏筹码，不可直接提现）
  
兑换：
  balance → chips（exchange_in）
  chips → balance（exchange_out）
```

**Prisma Schema 变更：**
```prisma
model Wallet {
  id            String   @id @default(uuid())
  userId        String   @unique
  balance       Float    @default(0)   // 可提现余额
  chips         Float    @default(0)   // 游戏筹码
  frozenChips   Float    @default(0)   // 坐桌时冻结的筹码
  user          User     @relation(fields: [userId], references: [id])
}
```

**比例设计（参考推荐）：**
```
1 USDT = 1000 Chips
小盲 = 10 Chips = 0.01 USDT
大盲 = 20 Chips = 0.02 USDT
```

#### 7. Tick Loop 重构

**问题：** 当前操作直接执行，存在并发竞争风险。

**改造方案：**
```typescript
class Table {
  private actionQueue: Action[] = [];
  
  enqueueAction(action: Action): void {
    this.actionQueue.push(action);
  }
  
  tick(): void {
    while (this.actionQueue.length > 0) {
      const action = this.actionQueue.shift()!;
      this.applyAction(action);
    }
    this.checkTimers();
    // 只在有变化时广播（减少不必要的 emit）
    if (this.dirty) {
      this.broadcastState();
      this.dirty = false;
    }
  }
}

// 全局 Tick 调度
setInterval(() => {
  for (const table of tableManager.getAllTables()) {
    table.tick();
  }
}, 100); // 10 tick/s
```

### 🟢 P2 — 中长期规划（扩展性）

#### 8. USDT 充值链路

**方案：** TRC20（手续费低，适合 Web3 游戏）

**架构：**
```
用户申请充值地址
      ↓
系统分配独立充值地址（地址池）
      ↓
用户转账 USDT → 充值地址
      ↓
区块链监听服务确认到账（N个区块确认）
      ↓
user.balance += amount（记录 DEPOSIT transaction）
      ↓
用户手动兑换 balance → chips
```

**推荐库：** `tronweb` (TRC20) 或 `viem` (ERC20/BEP20)

#### 9. 防洗钱风控系统

**最小可行方案：**
```typescript
// 规则1：同桌频率
async checkCollusionRisk(playerA: string, playerB: string): Promise<boolean> {
  const count = await prisma.handAction.count({
    where: {
      hand: { players: { some: { userId: { in: [playerA, playerB] } } } },
      createdAt: { gte: new Date(Date.now() - 24 * 60 * 60 * 1000) },
    }
  });
  return count > 100; // 24小时内同桌超100局 → 风控
}

// 规则2：胜率异常（A输给B超过80%）
// 规则3：同IP限制（同IP不允许同桌）
```
